Back Pressure Features

Simulate load and test system resilience with rate limiting and chaos error injection. Rate limiting uses a token bucket algorithm, and chaos errors introduce random failures to mimic real-world issues.

Control Flow

      graph TD
        A[Receive Request] --> B{Rate Limit Check}
        B -->|Within Limit| C{Check Chaos}
        B -->|Exceeds Limit| D[Return 429]
        C -->|X-Echo-Chaos| E[Random Failure]
        C -->|X-Echo-Error| F[Specific Error]
        E -->|Fail| G[Return 500/502/503/504/408]
        E -->|Pass| H[Continue Processing]
        F --> G
        H --> I[Process Request]

Rate Limiting

Limits requests per second (RPS) with a token bucket algorithm, controlled via environment variables or headers.

  • Headers: X-Echo-Rate-Limit-RPS: 50, X-Echo-Rate-Limit-Burst: 100
  • Env: ECHO_RATE_LIMIT_RPS=50, ECHO_RATE_LIMIT_BURST=100
  • Example:
      
curl -H "X-Echo-Rate-Limit-RPS: 10" -H "X-Echo-Rate-Limit-Burst: 20" http://localhost:8080/
# If limit exceeded: HTTP 429 Too Many Requests

Chaos Errors

Inject random failures with a specified probability (0-100%).

  • Header: X-Echo-Chaos: 30 (30% chance of failure)
  • Env: ECHO_CHAOS=15
  • Errors: 500, 502, 503, 504, 408 (randomly chosen)
  • Example:
      
curl -H "X-Echo-Chaos: 30" http://localhost:8080/
# ~30% chance of HTTP 500/502/503/504/408
# Logs: "Chaos error injected: 503"

Specific Error Injection

Force a specific HTTP error code.

  • Header: X-Echo-Error: 503 (e.g., 500, 502, 503, 504, 408, 429)
  • Example:
      
curl -H "X-Echo-Error: 503" http://localhost:8080/
# Response: HTTP 503 Service Unavailable
# Logs: "Forced error: 503"

Notes

  • Rate limiting is skipped for SSE requests to ensure uninterrupted streaming.
  • Chaos errors are tracked via echo_chaos_errors_total Prometheus metric.
  • Headers override environment variables for per-request control.
